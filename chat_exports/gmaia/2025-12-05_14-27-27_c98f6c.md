---

## **What Horizon Provides: Modular Stack (Traditional Approach)**

Horizon's software architecture is **modular/rule-based**:

```
Sensors → Perception Module → Prediction Module → Planning Module → Control Module → Actuators
  ↓            ↓                    ↓                   ↓               ↓
Camera      Object Detection    Trajectory Pred    Path Planning   Steering/Braking
Radar       Lane Detection      Behavior Pred      Decision Rules   Acceleration
Lidar       Segmentation        Risk Assessment    Motion Planning  
```

**Key characteristics:**
- **Separate neural networks** for each task (detection, tracking, prediction, etc.)
- **Hand-crafted rules** connect modules (if-then logic, finite state machines)
- **Explicit intermediate representations** (bounding boxes, lane lines, occupancy grids)
- **Human-engineered features** and post-processing

**This is the "traditional" approach** used by most ADAS suppliers (Mobileye, Bosch, Continental).

---

## **What "End-to-End" Means: Single Neural Network**

True end-to-end systems (like **Tesla FSD v12+**, **Wayve**, **Waabi**) work differently:

```
Sensors → Single Large Neural Network → Control Outputs
  ↓              (Black Box)                ↓
Camera      Learns everything from      Steering angle
Radar       human driving data          Acceleration
Lidar                                   Braking
```

**Key characteristics:**
- **One neural network** maps sensor inputs directly to control outputs
- **No hand-crafted rules** (the network learns driving behavior from data)
- **No explicit intermediate representations** (no bounding boxes, just internal embeddings)
- **Data-driven** rather than engineering-driven

---

## **Why This Distinction Matters (Disruption Lens):**

### **Modular Approach (Horizon's Current State):**
✅ **Pros:**
- Easier to debug and validate (regulators prefer this)
- Lower data requirements (each module trained separately)
- Can mix-and-match components (flexibility)
- Predictable behavior in known scenarios

❌ **Cons:**
- **Scaling ceiling**: Performance plateaus because errors compound across modules
- **Long-tail problem**: Can't handle novel scenarios not explicitly programmed
- **Engineering bottleneck**: Requires manual tuning for every edge case
- **Slower improvement**: Adding capabilities requires human engineering work

### **End-to-End Approach (Disruption Pattern):**
✅ **Pros:**
- **Better scaling**: Performance improves with more data (no plateau)
- **Handles long-tail**: Learns from human demonstrations (any scenario humans can drive)
- **Faster iteration**: Just add more data, no manual engineering per edge case
- **Emergent capabilities**: Network discovers solutions humans didn't explicitly program

❌ **Cons:**
- **Black box**: Hard to explain why it makes decisions (regulatory challenge)
- **Massive data requirements**: Needs millions/billions of miles of driving data
- **Compute intensive**: Requires powerful training infrastructure
- **Risk of unexpected failures**: Can fail in non-intuitive ways
