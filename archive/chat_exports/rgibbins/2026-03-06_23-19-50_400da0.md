---

## 1. Data Ingestion & Monitoring Pipelines

A proactive intelligence engine requires continuous, scheduled ingestion across three signal classes:

**Class A — Scheduled Release Monitoring (Calendar-Anchored)**
- BLS Employment Situation: first Friday of each month, auto-fetch within 60 seconds of release
- METR benchmark publications: continuous scrape of metr.org and associated arXiv preprints
- BLS JOLTS, QCEW, OES: monthly/quarterly ingestion with sector-level parsing
- Federal Reserve H.8 (bank credit), Z.1 (financial accounts): quarterly
- Inference cost trackers: OpenAI, Anthropic, Google pricing APIs — daily delta checks

**Class B — Continuous Signal Monitoring (Threshold-Triggered)**
- AI inference cost per token: daily API price checks across major providers
- METR task-completion horizon: new benchmark publications trigger immediate ingestion
- Sector payroll deltas: month-over-month change by 3-digit NAICS code
- Job openings-to-hires ratio by sector: leading indicator of AI absorption vs. hiring freeze
- Robot installation data (IFR): quarterly, parsed by sector and region

**Class C — Anomaly Detection Feeds (Model-Baseline Comparison)**
- Actual payroll vs. displacement model projection: flag when divergence exceeds ±15%
- Sector employment vs. METR capability frontier: flag when sector tipping point is crossed
- AI cost curve vs. projected trajectory: flag when actual cost drops faster than model baseline

The ingestion layer is a standard ETL pipeline (fetch → parse → normalize → store → compare against model baselines). No exotic infrastructure required — the complexity is in the comparison logic, not the ingestion.

## 3. Training & Fine-Tuning for Proactive Behavior

Three approaches, in order of implementation complexity:

**Approach A — Scheduled Synthesis (Lowest Complexity)**
Scheduled jobs run the displacement skill, energy sector skill, and copper/battery skills on a fixed cadence (monthly for BLS-anchored skills, quarterly for structural updates). Output is a standing report comparing actuals to model projections. This requires no model fine-tuning — only orchestration logic.

**Approach B — Anomaly Detection Against Model Baselines (Medium Complexity)**
The system maintains a "model state" — the current projected trajectory for each key variable (displacement rate, AI cost, sector payrolls, METR horizon). Each new data ingestion computes the delta between actual and projected. When delta exceeds a defined threshold, the system flags it as an anomaly and generates an explanatory analysis. This is a comparison function, not a generative one — it can be implemented as a rule-based layer on top of the existing skill architecture.

**Approach C — Fine-Tuning for Push-Style Reporting (Highest Complexity)**
Fine-tuning the underlying model on examples of proactive analytical behavior: (a) ingesting a data release, (b) comparing to model baseline, (c) generating a structured push report without a user prompt. This requires a curated dataset of (data_input → analysis_output) pairs, where the analysis is grounded in the disruption framework. The training signal is: "given this data, what would the disruption framework conclude, unprompted?" Estimated fine-tuning dataset size: 500–2,000 high-quality examples across the skill domains.

Approaches A and B can be implemented immediately within the existing architecture. Approach C requires a 3–6 month fine-tuning cycle.

## 4. Tradeoffs: Fully Autonomous vs. Human-in-the-Loop

| Dimension | Fully Autonomous | Human-in-the-Loop |
|-----------|-----------------|-------------------|
| **Speed** | Seconds from data release to report | Minutes to hours (human confirmation delay) |
| **False positive rate** | Higher — threshold crossings can be noise | Lower — human filters spurious signals |
| **Analytical depth** | Consistent, model-grounded | Variable, depends on human judgment |
| **Missed signals** | Near-zero (systematic coverage) | Higher (human attention is finite) |
| **Trust calibration** | Requires track record to establish | Immediate, familiar workflow |
| **Compounding value** | Exponential — each report builds model state | Linear — each report is standalone |

**The correct architecture is not binary.** The optimal design is a tiered confirmation model:

- **Tier 1 (Fully Autonomous)**: Scheduled synthesis reports, model-state updates, anomaly flagging — no human confirmation required. These are deterministic outputs from known inputs.
- **Tier 2 (Human-Confirmed)**: Threshold-crossing alerts that trigger major analytical conclusions (e.g., "NFP has turned negative — initiating full displacement cascade analysis"). System generates the analysis; human confirms before distribution.
- **Tier 3 (Human-Initiated)**: Deep-dive investigations, scenario analysis, investment thesis construction — user-prompted, as today.

## Decision 1: Model State — What Gets Tracked and How

This is the foundational decision. Everything else depends on it.

The system needs a persistent "model state" — a structured representation of current projected trajectories for every key variable. Without this, there is no baseline to compare against, and the anomaly detection layer has nothing to trigger on.

**What must be in model state (minimum viable set):**
- Economy-wide FTE displacement rate (monthly, by scenario)
- AI cost per effective labor-hour (daily)
- METR task-completion horizon (updated on each new benchmark publication)
- Sector payroll projections by 3-digit NAICS (monthly)
- AI inference cost per million tokens by provider (daily)

**The decision:** Who owns model state updates? Three options:
1. **Skill-driven**: Each skill run updates its own slice of model state automatically
2. **Human-curated**: A human reviews and approves model state updates before they become the new baseline
3. **Hybrid**: Skills update automatically within defined bounds; human approval required for parameter changes (e.g., changing METR doubling time from 7 months to 5 months)

Option 3 is the correct answer. Automatic updates within bounds prevent model drift from noise; human approval gates prevent a single anomalous data point from corrupting the baseline.

---

## Decision 2: Threshold Definitions — Model-Derived vs. Statistical

The thresholds in the architecture proposal are model-derived (e.g., NFP crosses zero, displacement rate crosses 5%). The alternative is statistical (e.g., NFP is 2 standard deviations below trailing 12-month mean).

**Model-derived thresholds** are correct for this system. Statistical thresholds detect anomalies relative to recent history — they would have flagged the -92,000 print as unusual, but they cannot tell you *why* it matters or what it implies for the displacement trajectory. Model-derived thresholds fire when the data crosses a theoretically meaningful boundary.

## Decision 3: Push Report Format — Structured vs. Narrative

When a threshold fires, the system generates a push report. Two format options:

**Structured (machine-readable first):** JSON payload with key metrics, delta from model, triggered threshold, and a short narrative summary. Designed for downstream consumption — dashboards, alerts, portfolio systems.

**Narrative (human-readable first):** Full analytical report in the format of the February 2026 analysis above — thesis, data table, model comparison, sector cascade, conclusion. Designed for direct human consumption.

The correct answer depends on the end consumer. If this system feeds a portfolio management workflow, structured output is the right primary format with narrative as a secondary layer. If the end consumer is a human analyst or decision-maker, narrative-first is correct.

**The decision:** Define the primary consumer of push reports before building the output layer. This determines the entire downstream architecture.

---

## Decision 4: Skill Invocation Cadence — Event-Driven vs. Scheduled

The skills (artificial labor, energy sector, copper, lithium-ion) are computationally intensive. Two invocation models:

**Event-driven:** Skills run only when a threshold is crossed. Lower compute cost, but model state can go stale between events.

**Scheduled:** Skills run on a fixed cadence (monthly for labor, quarterly for energy/copper/batteries) regardless of threshold crossings. Higher compute cost, but model state is always current.

**The correct architecture is both:** Scheduled runs maintain current model state. Event-driven runs are triggered by threshold crossings and use the current model state as context. The scheduled run is the heartbeat; the event-driven run is the alert.

**The decision:** Set the scheduled cadence for each skill. Proposed defaults:
- Artificial labor: monthly (anchored to BLS release cycle)
- Energy sector: quarterly (anchored to EIA/grid data release cycle)
- Copper forecast: quarterly
- Lithium-ion: quarterly

---

Approach C (fine-tuning) from the architecture proposal is the highest-complexity, highest-value long-term investment. But it is not the first investment.

**The correct sequencing:**
1. Build Approaches A and B first (orchestration + anomaly detection) — these deliver 80% of the proactive value with 20% of the effort
2. Use the outputs of A and B to generate the fine-tuning training dataset — every push report produced by the orchestration layer is a (data_input → analysis_output) pair
3. Fine-tune on the accumulated dataset after 6–12 months of operation

This means the fine-tuning dataset builds itself through normal operation. You do not need to curate 2,000 examples manually — you run the system, generate reports, review them for quality, and the approved reports become training data.

**The decision:** Commit to the A→B→C sequencing now. The temptation will be to skip to fine-tuning because it sounds like the "real" solution. It is not — orchestration is the real solution. Fine-tuning is the optimization layer on top of a working system.
